# Berry on Propeller 2 v0.9.1

This release adds the first working native Berry `i2c` and `spi` modules on the Propeller 2 Catalina path, while keeping the existing REPL, standard-library, filesystem, and `prop2_*` helper work intact.

## Release Binaries

This release ships two Catalina binaries:

- `berry_p2.binary`
  The normal Berry application image for RAM loading.
- `berry_p2_flash_loader.binary`
  The Catalina flash-programmer wrapper image generated from `berry_p2.binary`.

## Load to RAM or Flash

### Load to RAM

Use the normal Berry application image for an interactive RAM session:

```sh
make p2 TOOLCHAIN=catalina
make p2-run TOOLCHAIN=catalina PORT=/dev/cu.usbserial-P97cvdxp
```

You can also use:

```sh
make p2-ram
```

once you have run `make configure`.

### Flash to SPI

Use the Catalina flash wrapper image for flash installs:

```sh
make p2 TOOLCHAIN=catalina
make p2-flash TOOLCHAIN=catalina PORT=/dev/cu.usbserial-P97cvdxp
tio -b 230400 /dev/cu.usbserial-P97cvdxp
```

Important notes:

- `berry_p2.binary` is the app image
- `berry_p2_flash_loader.binary` is the temporary flash-programmer wrapper
- on the Catalina path, do not use direct `loadp2 -SPI build/p2/catalina/berry_p2.binary`
- for P2 Edge development with serial attach after reset, use `FLASH=ON, △=OFF, ▽=OFF`
- for flash-only fast boot, use `FLASH=ON, △=OFF, ▽=ON`

## New Features

### Native `i2c` Module

The port now includes a native Berry `i2c` module for simple sensor and peripheral work. It keeps one module-global bus configuration so it is easy to use from the REPL and small scripts.

Supported API:

- `i2c.init(scl_pin, sda_pin, khz)`
- `i2c.write(addr, data)`
- `i2c.read(addr, count)`
- `i2c.writeread(addr, txdata, rxcount)`
- `i2c.scan()`

Example:

```berry
import i2c

i2c.init(25, 24, 100)
print(i2c.scan())
print(i2c.writeread(0x77, "\xD0", 1))
```

Live hardware verification on the current P2 Edge setup:

- `i2c.scan()` returns `[119]`
- `119` is `0x77`, the BMP180 address
- `i2c.writeread(0x77, "\xD0", 1)` returns `U`, which is byte `0x55`, the BMP180 chip ID

### Native `spi` Module

The port now includes a native Berry `spi` module with explicit chip select control, mode selection, write/read helpers, and full-duplex transfer.

Supported API:

- `spi.init(clk_pin, mosi_pin, miso_pin, cs_pin, mode, khz)`
- `spi.select()`
- `spi.deselect()`
- `spi.write(data)`
- `spi.read(count, filler=0xFF)`
- `spi.transfer(data)`

Example:

```berry
import spi

spi.init(10, 11, 12, 13, 0, 1000)
spi.select()
id = spi.transfer("\x9F\x00\x00\x00")
spi.deselect()
print(id)
```

This is intended for common flash, EEPROM, ADC, and sensor bring-up.

## Existing Features

### Interactive Berry REPL

The P2 port already supports a usable Berry REPL for interactive testing and board bring-up.

Example:

```berry
print(1 + 2)
a = 6
print(a * 7)
for i:0..3
    print(i)
end
```

### Standard Modules: `string`, `math`, `json`

The port already supports a useful set of standard modules, so scripts can do formatting, math, and JSON parsing without dropping into C.

Example:

```berry
import string
import math
import json

print(string.toupper("berry"))
print(math.sqrt(81))
print(json.dump({"a":1,"b":[2,3]}))
```

### Raw Byte Work with `bytes`

The `bytes` support is already useful for buffers, binary payloads, and file content inspection.

Example:

```berry
b = bytes().fromstring("AB")
print(b.tohex())
print(bytes("1122"))
```

### SD Card Files and `os`

The port already supports SD-backed file I/O and a practical subset of `os`, including directory listing and path helpers.

Example:

```berry
import os

print(os.listdir("/"))
f = open("/TMPD/TEST.TXT", "w")
f.write("hello from berry")
f.close()
print(open("/TMPD/TEST.TXT", "r").read())
```

### `prop2_*` Hardware Helpers

The port already exposes low-level Propeller 2 helpers as `prop2_*` globals for clocks, counters, pins, smartpins, locks, CORDIC, and cog management.

Example:

```berry
print(prop2_clock_freq())
print(prop2_ticks())
prop2_pin_output(56)
prop2_pin_write(56, 1)
print(prop2_pin_read(56))
```

### Catalina RAM and Flash Workflow

The Catalina workflow already separates RAM loading from flash programming cleanly, so the normal development path is easy to repeat.

Example:

```sh
make configure TOOLCHAIN=catalina PORT=/dev/cu.usbserial-P97cvdxp \
  P2_SILICON=latest CATALINA_PLATFORM=P2_EDGE CATALINA_MODEL=COMPACT

make p2-ram
make p2-flash
make p2-attach
```

## Notes

- The current live-verified hardware path is macOS + Catalina + `P2_EDGE` + latest silicon
- The verified serial port on the current setup is `/dev/cu.usbserial-P97cvdxp`
- The current terminal baud for the Catalina `P2_EDGE` path is `230400`
- `i2c` was live-verified against a BMP180 on `SCL=25`, `SDA=24`
- `spi` import and initialization were verified on hardware; full device-level SPI transfer validation still depends on the attached target device
