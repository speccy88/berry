# Berry on Propeller 2 v0.9.3

This release turns the P2 fork into a broader hardware bring-up build: ergonomic `p2` GPIO helpers, the first second-cog Berry worker, lightweight `threads` channels, expanded I2C/SPI APIs, and a first Spin2 loader/tooling path. It also fixes the Catalina/P2 native module loading issue that caused imports or native calls to hang.

## Release Binaries

This release ships two Catalina binaries:

- `berry_p2.binary`
  The normal Berry application image for RAM loading.
- `berry_p2_flash_loader.binary`
  The Catalina flash-programmer wrapper image generated from `berry_p2.binary`.

## Load to RAM or Flash

Load to RAM while developing:

```sh
make p2 TOOLCHAIN=catalina
make p2-run TOOLCHAIN=catalina PORT=/dev/cu.usbserial-P97cvdxp
```

Flash to the P2 Edge SPI flash:

```sh
make p2 TOOLCHAIN=catalina
make p2-flash TOOLCHAIN=catalina PORT=/dev/cu.usbserial-P97cvdxp
tio -b 230400 /dev/cu.usbserial-P97cvdxp
```

Important notes:

- Use `berry_p2.binary` for RAM loading.
- Use `berry_p2_flash_loader.binary` for Catalina flash programming.
- Do not flash `berry_p2.binary` directly with `loadp2 -SPI` on the Catalina path.
- For P2 Edge development with serial attach after reset, use `FLASH=ON, △=OFF, ▽=OFF`.

## New Features

### `p2` Hardware Module

The `p2` module wraps common P2 hardware operations in a Berry-friendly namespace. It includes GPIO, delays, cog ID, cog stop/state helpers, locks, heap reporting, and a small `beep` helper.

```berry
import p2

p2.pinmode(56, p2.OUTPUT)
p2.high(56)
p2.sleep_ms(250)
p2.low(56)
print(p2.cogid())
print(p2.sbrk())
```

### Second-Cog Berry Worker

`worker.start()` launches a second cog with its own Berry VM and Hub-RAM heap. `worker.exec(name, ...int_args)` sends a name-based mailbox request; v1 supports integer arguments only.

```berry
import worker

print(worker.start())
worker.exec("blink", 56, 250)
print(worker.state())
worker.stop()
```

### `p2.cog_start()` Convenience

`p2.cog_start(name, ...int_args)` is a higher-level convenience wrapper over the worker backend. It keeps v1 safe by dispatching by function name instead of transferring closures between VMs.

```berry
import p2
import worker

cog = p2.cog_start("blink", 56, 250)
print(cog)
print(worker.state())
worker.stop()
```

### `threads` Channels

The `threads` module starts the Catalina-Lua-inspired concurrency surface with fixed Hub-RAM channels and a tiny shared key/value store. Values are integers or strings in v1.

```berry
import threads

threads.channel("sensor")
threads.put("sensor", 123)
print(threads.get("sensor"))
threads.update("last", "ok")
print(threads.value("last"))
```

### Expanded `i2c` Module

The I2C module now includes `present`, `wait`, optional pullup parameter storage, and low-level `start`/`stop` helpers, while keeping the important `scan()` path simple.

```berry
import i2c

i2c.init(25, 24, 400)
print(i2c.scan())                         # [119] with the tested BMP180
print(i2c.present(0x77))                  # true
print(i2c.writeread(0x77, "\xD0", 1))     # "U" == 0x55 chip id
```

### Expanded `spi` Module

SPI keeps the v1 module-global bus state and now includes `stop()` to release the configured pins. Transfers return raw Berry strings.

```berry
import spi

spi.init(10, 11, 12, 13, 0, 1000)
spi.select()
id = spi.transfer("\x9F\x00\x00\x00")
spi.deselect()
print(id)
spi.stop()
```

### `spin2` Loader Prototype

The `spin2` module can list binaries on SD, load a compatible binary into Hub RAM, start it in a cog, exchange integer calls through a mailbox, and stop it. The repo also has a `spin2/` source directory and `make spin2` build target.

```berry
import spin2

print(spin2.list())
handle = spin2.start("berry_mailbox_demo.bin")
print(spin2.call(handle, 1, 123))
spin2.stop(handle)
```

## Existing Features

### Interactive Berry REPL

The P2 port still supports a practical interactive REPL for quick experiments and board bring-up.

```berry
print(1 + 2)
a = 6
print(a * 7)
```

### Standard Modules

Core modules such as `string`, `math`, and `json` remain available.

```berry
import string
import math
import json

print(string.toupper("berry"))
print(math.sqrt(81))
print(json.dump({"a": 1, "b": [2, 3]}))
```

### Binary Data with `bytes`

`bytes` is useful for packets, register payloads, and file data.

```berry
b = bytes().fromstring("AB")
print(b.tohex())
```

### SD Files and `os`

The P2 build keeps SD-backed file and `os` helpers available when the SD card is mounted.

```berry
import os

print(os.listdir("/"))
f = open("/TMPD/TEST.TXT", "w")
f.write("hello from berry")
f.close()
```

### Low-Level `prop2_*` Helpers

The lower-level P2 globals remain available for direct clock, pin, smart-pin, lock, CORDIC, and cog experiments.

```berry
print(prop2_clock_freq())
prop2_pin_output(56)
prop2_pin_write(56, 1)
print(prop2_pin_read(56))
```

### Catalina and FlexC Build Paths

Catalina is the verified release path, and FlexC remains part of the build model.

```sh
make p2 TOOLCHAIN=flexc
make p2 TOOLCHAIN=catalina
```

## Verification Notes

- Built with Catalina 8.8.9 through Docker.
- RAM-loaded on P2 Edge using `/dev/cu.usbserial-P97cvdxp`.
- Verified `p2.cogid()` returns `0`.
- Verified pin 56 low/high readback as `0` and `1`.
- Verified BMP180 on `SCL=25`, `SDA=24`: `i2c.scan()` returned `[119]`, `i2c.present(0x77)` returned `true`, and register `0xD0` returned `U`.
- Verified `worker.start()` returned cog `5`, `worker.exec("noop", 7)` returned the worker to `ready`, and `worker.exec("blink", 56, 50)` entered `running` until `worker.stop()`.
- Verified `threads.channel` / `put` / `get` with integer values.
- Verified `spin2` import, `spin2.path()`, and empty-list behavior when `/spin2` has no binaries.
- Verified `make spin2` builds `spin2/build/berry_mailbox_demo.bin`.

## V1 Limitations

- Worker and `p2.cog_start()` dispatch only by function name and integer arguments.
- The worker VM has its own environment; it cannot see closures or objects from the main VM.
- `threads` is a small channel/key-value layer, not a full scheduler yet.
- `spin2.call()` requires binaries that implement the documented Berry mailbox convention.
- SPI device-level JEDEC validation still depends on the attached SPI target.
